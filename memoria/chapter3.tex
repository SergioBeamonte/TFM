\chapter{Implementation}
\label{ch:implementation}

This chapter describes the proof-of-concept implementation used to
estimate the parameter vector $\boldsymbol{\Theta}$ of a regular
Influence Diagram from a partial expert rule base $\mathcal{R}$.
The system is built on top of three external components: the
\textsc{SMILE} reasoning engine and its \texttt{pysmile} bindings,
which provide the \texttt{.xdsl} parser and the Shachter evaluation
algorithm~\cite{shachter1986evaluating,ezawa1994evidence,genie};
\texttt{NumPy} for the in-memory representation of the CPTs; and
\textsc{EDAspy}, the open-source library of Soloviev et
al.~\cite{soloviev2024edaspy} that implements the EDA variants
discussed in Chapter~\ref{ch:framework}. Around these dependencies the
system is organised in four functional layers:

\begin{enumerate}
    \item \textbf{Model loading and structural extraction.}
          The \texttt{.xdsl} file is parsed into an internal node
          dictionary that holds the graph structure and a
          \texttt{NumPy} reference to every numerical table.
    \item \textbf{Rule-base parsing.}
          The expert rule base is read from a CSV file and translated
          into a list of $(\mathbf{evidence},\, D_{i_l},\, a^{(l)})$
          triples.
    \item \textbf{Fitness evaluation.}
          For each candidate parameter vector, the Shachter engine is
          invoked once per rule and the per-rule losses are aggregated
          into the score the EDA tries to minimise.
    \item \textbf{EDA-based optimisation.}
          The continuous UMDA implementation of \textsc{EDAspy} drives
          the search over an encoded parameter vector
          $\boldsymbol{\phi}$.
\end{enumerate}

% ==============================================================
\section{Model Loading and Structural Extraction}
% ==============================================================

The input Influence Diagram is provided in the \texttt{.xdsl} format,
the native format of \textsc{GeNIe}/\textsc{SMILE}~\cite{genie}. The
\texttt{pysmile} bindings expose the parsed network and the
\texttt{NetworkExtractor} module walks over it once to build the node
dictionary $\mathcal{N}$: for every node it records the kind (chance,
decision or utility), the ordered list of parents, the state labels and
a \texttt{NumPy} reference to its numerical table.

Storing a reference rather than a copy lets the decoder write the
updated values back in place before each fitness evaluation, without
rebuilding the SMILE network every time. This matters because the
Shachter evaluation is by far the most expensive operation of the
pipeline, so it is reused as much as possible.

% ==============================================================
\section{Rule-Base Parsing}
% ==============================================================

The expert rule base $\mathcal{R}$ is delivered as a CSV file with one
rule per row and one column per node of the diagram. Each row records,
for the variables that play a role in that rule, their observed states,
and for the target decision the action the expert considers optimal; a
sign convention distinguishes evidence from the target, and marks the
nodes that are not informational parents of the decision as irrelevant.
Rows without a valid target are reported and discarded. The output of
this stage is the list of \emph{(evidence, decision, expected action)}
triples consumed by the fitness function.

% ==============================================================
\section{Parameter Encoding}
% ==============================================================

The EDA searches over plain real vectors, but the parameters of an ID
are constrained: every CPT row must be a probability distribution and
the utilities must stay on a bounded scale. The encoding separates the
two concerns. The optimiser works on a \emph{raw} vector
$\boldsymbol{\phi}$ that it samples freely inside a box, and a
\emph{decoder} turns that raw vector into a legal model. A fixed node
ordering tells the decoder which slice of $\boldsymbol{\phi}$ belongs
to each table.

\subsection{Encoded vector}

All the free entries of the CPTs and of the utility table are
concatenated into a single vector $\boldsymbol{\phi}\in\mathbb{R}^{d}$;
\textsc{EDAspy} samples each dimension within user-supplied bounds, in
our runs $[-5,5]$ for the CPT entries and $[-10,10]$ for the utilities.
The bounds are symmetric around zero on purpose: combined with the
squashing functions of the next two subsections, the raw zero decodes
to a uniform CPT row and to the mid-range utility, so the search starts
with no built-in preference for any state of a chance variable or any
action of a decision.

\subsection{Decoding chance nodes}

Each CPT row is recovered by passing the corresponding raw entries
through a softmax with temperature $T_c$,
\[
   \hat{\theta}(\cdot \mid \mathbf{pa}) =
   \operatorname{softmax}\!\bigl(\boldsymbol{\phi}_{\text{row}} / T_c\bigr),
\]
which always returns non-negative values that sum to one. Two
consequences follow. First, there is \emph{no repair step}: the
optimiser may roam an unconstrained real space and the decoded model is
always legal, so the Shachter engine can be called unconditionally.
Second, the temperature controls how peaked the decoded distribution
is: a low $T_c$ pushes the row towards a near-deterministic
distribution, a high $T_c$ flattens it towards the uniform.

\subsection{Decoding the utility node}

Each free utility entry is mapped through a sigmoid and rescaled to the
target range $[u_{\min},u_{\max}]$ (here $[0,10]$),
\[
   \hat{u} = \sigma(\phi / T_u)\,(u_{\max} - u_{\min}) + u_{\min},
\]
followed by two hard overrides: $\hat{u}(\mathbf{pa}^{+}) = u_{\max}$
and $\hat{u}(\mathbf{pa}^{-}) = u_{\min}$, where $\mathbf{pa}^{+}$ and
$\mathbf{pa}^{-}$ are the best and worst configurations declared by the
expert at initialisation.

The sigmoid is monotone, so the \emph{ordering} of the raw utility
values is preserved in the decoded utilities. This is what matters,
because the optimal policy depends only on that ordering; its
saturation means that most of the sensitivity is concentrated near the
midpoint, while extreme raw values barely move the decoded utility. The
two pinned entries are not cosmetic: because $u$ is identifiable only
up to a positive affine transformation, fixing the best and worst
outcomes removes that ambiguity and stops the EDA from drifting along a
direction that does not change the policy. The same
partial-identifiability phenomenon is studied formally by Lazzati and
Metelli~\cite{lazzati2025utilities} for utility learning in MDPs.

\begin{proposition}[Validity of the decoded model]
\label{prop:valid-cpt}
For every raw vector $\boldsymbol{\phi}$ the decoded tables form a
valid parameterisation: softmax guarantees that each CPT row is
non-negative and sums to one, and the rescaled sigmoid keeps every
utility inside $[u_{\min},u_{\max}]$. The Shachter engine can therefore
be evaluated unconditionally.
\end{proposition}

\section{Fitness Function}
% ==============================================================

A candidate is scored by how well the policy it induces reproduces the
expert rules. This is the mean per-rule loss
$\mathcal{L}(\boldsymbol{\Theta})$ of
Definition~\ref{def:fitness}, which \textsc{EDAspy} minimises directly.
A central point is that this loss is \emph{not unique}: the per-rule
term $\ell$ is a configurable choice, and the implementation exposes
the whole family introduced in Chapter~\ref{ch:framework} — the
$0$--$1$ \textsc{binary} loss, \textsc{regret}, \textsc{margin},
\textsc{softmax} and \textsc{entropy} — whose effect on convergence and
final accuracy is compared experimentally in Chapter~4. The simplest
member is the $0$--$1$ loss, whose mean is just the fraction of rules
the model gets wrong:

\begin{definition}[Rule-accuracy cost]
\label{def:cost}
With the $0$--$1$ per-rule loss, the objective reduces to
\[
   c(\boldsymbol{\phi}) = 1 -
   \frac{1}{|\mathcal{R}|}
   \sum_{(\mathbf{x}^{(l)}, D_{i_l}, a^{(l)})\in\mathcal{R}}
   \mathbb{1}\!\left[\delta^{*}_{i_l}\!\left(\boldsymbol{\Theta}
   (\boldsymbol{\phi}),\mathbf{x}^{(l)}\right) = a^{(l)}\right],
\]
where $\boldsymbol{\Theta}(\boldsymbol{\phi})$ is the decoded vector
and $\delta^{*}_{i_l}$ is the action returned by the Shachter engine
under the evidence $\mathbf{x}^{(l)}$. The smoother variants replace
the indicator by the corresponding $\ell$ of
Chapter~\ref{ch:framework}, exposing more gradient-like information to
the search.
\end{definition}

Whatever the loss, evaluating one candidate means decoding it into the
tables and, for each rule, entering its evidence, running the diagram
and inspecting the recommended action. Degenerate tables sampled in
the early generations can occasionally make the engine fail
numerically; those rules are simply skipped rather than penalised,
since the share of unevaluable candidates vanishes quickly as the
search progresses.

\section{The EDAspy Library and the Choice of UMDA\textsubscript{c}}
% ==============================================================

\textsc{EDAspy}~\cite{soloviev2024edaspy} is the reference open-source
EDA library of the UPM CIG group; it implements the univariate and
multivariate variants reviewed in Chapter~\ref{ch:framework} behind a
common interface. This work uses its continuous UMDA (\texttt{UMDAc})
for three reasons:
\begin{itemize}
  \item the search space is high-dimensional ($d>10^{3}$ in the
        benchmark of Chapter~4), which makes the full covariance
        estimation of richer variants such as EGNA expensive;
  \item the validity of the decoded model (Proposition~\ref{prop:valid-cpt})
        is enforced row by row, so modelling cross-parameter
        dependencies brings little benefit here;
  \item a univariate model has a single hyperparameter per dimension,
        keeping the tuning of Chapter~4 tractable.
\end{itemize}
Richer variants, SPEDA in particular, are left to the perspectives
chapter.

\subsection{UMDA\textsubscript{c} internal model}

UMDA\textsubscript{c} maintains, for each dimension, an independent
Gaussian marginal whose mean and variance are refitted every generation
from the sample moments of the selected elite. The mechanics were given
in Chapter~\ref{ch:framework}; here we only note that the model is
isotropic across dimensions and that its variance is what eventually
contracts to mark convergence.

\subsection{Hyperparameters}

The run is governed by the few hyperparameters of
Table~\ref{tab:hyperparams}. The search stops early when the best cost
does not improve for $T_{\text{dead}}$ consecutive generations, which
avoids wasting time once the population has converged.

\subsection{Full optimisation loop}

The first population is drawn \emph{uniformly} within the symmetric raw
bounds. This is deliberate. Because the bounds are centred at zero and
both squashing functions are symmetric about it, a raw value of zero
decodes to a uniform CPT row and to the mid-range utility: the search
therefore starts \emph{unbiased}, with no built-in preference for any
state or action. Uniform sampling also scatters the initial individuals
across the whole raw space rather than around a single guess, so the
EDA explores broadly before its model contracts — which matters on the
multimodal, degenerate landscape of this problem, where many parameter
vectors reproduce the same rules and an early commitment to one region
would be premature.

From there on, each generation samples a new population from the
current Gaussian marginals, scores every candidate as described in the
previous section, selects the top-$\mu$ elite by truncation and refits
the marginals; the loop terminates when either the maximum number of
generations or the early-stopping criterion is reached. Two natural
extensions, both supported by \textsc{EDAspy} and discussed in the
perspectives chapter, replace this uninformed start or speed it up: an
LLM-generated warm start~\cite{nafar2025llm4bn} and parallel fitness
evaluation~\cite{mendiburu2005parallel}. Neither is used here, to keep
the experiments single-threaded and reproducible.

% ==============================================================
\section{Summary}
% ==============================================================

The pipeline turns the formal problem of Chapter~\ref{ch:framework}
into an executable loop: the diagram and the expert rules are loaded
once, a raw candidate vector is decoded through softmax and sigmoid
into a valid model (Proposition~\ref{prop:valid-cpt}), the diagram is
evaluated against every rule to produce the loss of
Definition~\ref{def:fitness} (in its simplest form the rule-accuracy
cost of Definition~\ref{def:cost}), and the continuous UMDA of
\textsc{EDAspy} searches for the vector of minimum cost. The next
chapter puts this pipeline to work on the neonatal-jaundice benchmark
of G{\'o}mez et al.~\cite{gomez2007jaundice}.
